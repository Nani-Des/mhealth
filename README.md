# mhealth

Cross-platform Flutter client for remote healthcare: bookings, referrals, chat, location-based provider search, notifications, and optional on-device symptom inference.

Overview
- mhealth is a multi-platform Flutter app that integrates Firebase (Auth, Firestore, Realtime DB, Storage, Functions, FCM) with local persistence (Hive), maps/geolocation, WebRTC/Agora for real-time calls, and on-device ML via TFLite.
- The app entry point is `lib/main.dart`. Key modules live under `lib/` (Auth, ChatModule, Appointments, Home, Maps, Services).
- Runtime features: background FCM handling, Remote Config-driven keys (Google Maps API), platform MethodChannels for maps/notifications, and an optional on-device TFLite model at `assets/models/disease_model.tflite`.
- Debug flags in `lib/main.dart` let you run UI-only sessions without initializing Firebase or platform services.

Tech stack
- Dart / Flutter (multi-platform)
- Firebase: Auth, Firestore, Realtime Database, Storage, Functions, Messaging, Remote Config
- Local persistence: Hive
- On-device ML: tflite_flutter + firebase_ml_model_downloader
- RTC / media: flutter_webrtc, agora_rtc_engine, just_audio, flutter_sound
- Networking: dio, http
- Maps & geolocation: google_maps_flutter, geolocator

Quick start (run locally)
Prerequisites
- Flutter SDK (Dart >= 3.5)
- Android SDK or Xcode (to build for mobile)
- Firebase project configured for your platforms
- Google Maps API key (if using maps)
- Optional: Agora credentials, Apple Sign-In .p8 key (keep private)

Steps
1. Clone and fetch packages
```
git clone https://github.com/Nani-Des/mhealth.git
cd mhealth
flutter pub get
```
2. Configure Firebase (one of)
- Using FlutterFire CLI:
```
flutterfire configure
```
- Or add platform config files:
```
android/app/google-services.json
ios/Runner/GoogleService-Info.plist
```
Ensure `firebase_options.dart` exists (generated by flutterfire) or create it accordingly.

3. Provide runtime secrets / keys
- Preferred: use Firebase Remote Config for Google Maps API and other runtime keys.
- Or use environment files (the project references `.env` via `flutter_dotenv`):
```
GOOGLE_MAPS_API_KEY=...
AGORA_APP_ID=...
AGORA_TOKEN=...
APPLE_P8_PATH=/path/to/AuthKey_XXXX.p8   # keep out of version control
```
Do not commit private keys. The repo currently contains a `.p8` file ã remove it and rotate the key if it is sensitive.

4. Emulators / functions (optional)
```
cd functions
npm install
firebase emulators:start --only firestore,auth,functions,database
```

5. Run the app
- On an Android emulator:
```
flutter run -d emulator-5554
```
- On web:
```
flutter run -d chrome
```
- To run quickly without Firebase (UI-only dev), set the debug flags at the top of `lib/main.dart`:
```dart
const bool debugDisableFirebase = true;
const bool debugDisableServices = true;
```

6. Build release artifacts
```
flutter build apk
flutter build ios
flutter build web
```

Key runtime details (what to inspect in the code)
- Entrypoint and init sequence: `lib/main.dart`
  - Firebase initialization (uses `DefaultFirebaseOptions.currentPlatform`)
  - Hive initialization and opening box `translations`
  - ConfigService initialization to fetch Remote Config keys (e.g., Google Maps API)
  - MethodChannel names:
    - `com.mhealth.nhap/maps` ã used to send Google Maps API key to native code
    - `com.mhealth.nhap/notifications` ã notification-related method calls (`updateFcmToken`, `handleNotification`)
  - Background FCM handler:
    - `_firebaseMessagingBackgroundHandler` stores pending notifications to SharedPreferences
    - Annotated with `@pragma('vm:entry-point')` so it runs in background isolates
  - FCM setup:
    - Requests notification permissions, retrieves token (with retry loop), saves token to Firestore `Users/<uid>.fcmToken`
    - Handles initial message and `onMessageOpenedApp` to navigate to booking screens
- Services referenced:
  - `ConfigService` ã reads Remote Config and exposes `googleMapsApiKey`
  - `CallService`, `WordFilterService` ã optional runtime services; controlled by debug flags
- On-device ML:
  - `assets/models/disease_model.tflite` and classes JSON files wired to `tflite_flutter` and `firebase_ml_model_downloader`
- Image cache:
  - Configured in `main.dart` to 100 MB via `imageCache.maximumSizeBytes`
- Navigation / state:
  - MultiProvider at app root with `AuthService` and `UserModel` (see `lib/` modules)
  - Splash animation in `CustomTransitionScreen` navigates to `HomePage`

Configuration checklist
- `firebase_options.dart` present and correct for target platforms
- Platform Firebase files in place (Android/iOS)
- Google Maps API key available via Remote Config or `.env`
- Agora and Apple Sign-In keys configured if using RTC / Sign-in
- Remove any private keys accidentally committed and rotate them if needed

Development tips
- Use `debugDisableFirebase` to iterate UI without backend.
- Remote Config keys are read at startup by `ConfigService`; hot reload will not re-fetch Remote Config unless you reinitialize it.
- Background FCM handling uses SharedPreferences to persist messages until the UI reads them.
- To debug native interactions, inspect the native platform code that listens to the same MethodChannel names.
- TFLite inference requires the model file at `assets/models/disease_model.tflite`; if you want remote model delivery, enable Firebase ML Model Downloader.

Testing
- Run unit and widget tests:

- Use Firebase emulators when testing functions or Firestore-heavy flows:
```
firebase emulators:start --only firestore,auth,functions,database
```
